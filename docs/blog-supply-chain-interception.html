Supply Chain Security

The Last Mile Problem:
Why 100% Coverage
is Hard

Every time a developer runs npm install, they're implicitly trusting hundreds of strangers. Intercepting that trust is straightforward in theory — and surprisingly subtle in practice.

May 2026 14 min read macOS · Linux

Supply chain attacks have gone from theoretical concern to recurring headline. SolarWinds compromised 18,000 organisations through a single build step. xz-utils nearly backdoored OpenSSH across every major Linux distribution. event-stream stole cryptocurrency from a specific npm package by becoming a dependency of a dependency. The colors.js and left-pad incidents showed that even intent-to-harm isn't required — a maintainer having a bad day is enough.

The attack surface is the package registry. Your code pulls from npm, PyPI, crates.io, pkg.go.dev, Maven Central, NuGet, Packagist. Each one is a trust decision you make dozens of times a day without thinking about it.

The question isn't whether to intercept package traffic. It's how to do it without breaking the development experience — and how close to 100% you can actually get.

This post documents what we learned building escrow, a local package proxy that applies age gates and OSV vulnerability scanning before allowing packages through. We tried every interception approach available on macOS and Linux, and each one has a ceiling.

The Trust Chain You're Defending

Before picking an interception strategy, it helps to understand what you're actually sitting in front of. When a developer runs npm install lodash, the chain looks like this: 

developer
  └─ npm CLI (resolves ~/.npmrc for registry URL)
       └─ DNS lookup: registry.npmjs.org → 104.16.x.34 (Cloudflare CDN)
            └─ TLS handshake + HTTP/1.1 GET /lodash
                 └─ tarball download → local cache → node_modules/

An interception layer can sit at any of these steps: the tool config, the DNS resolution, the TCP connection, or the TLS layer. The catch is that each insertion point has different coverage, different reliability, and different compatibility with the rest of a developer's environment — particularly VPNs.

The Options

Option 1 — Most Reliable
Per-tool registry config
Write the proxy URL directly into each tool's config file. npm uses ~/.npmrc, pip uses ~/.pip/pip.conf, cargo uses ~/.cargo/config.toml, Go uses GOPROXY, and so on. All seven major ecosystems support this natively.
Works
  • No kernel involvement
  • Works with any VPN
  • Hostname-aware naturally
  • Zero IP rotation issues
  • Standard proxy protocol
Doesn't catch
  • Bundled runtimes (VSCode)
  • GUI apps (no shell env)
  • Hardcoded registry URLs
  • Tools not in the list
Option 2 — GUI App Coverage
launchctl environment injection
On macOS, apps launched from the Dock or Spotlight don't inherit shell environment variables. A LaunchDaemon can inject env vars into the macOS launch environment at boot, so every process — including VSCode and Zed — inherits NPM_CONFIG_REGISTRY, GOPROXY, PIP_INDEX_URL, etc. 
# /Library/LaunchDaemons/com.escrow.environment.plist
# RunAtLoad — fires before login window
launchctl setenv NPM_CONFIG_REGISTRY http://127.0.0.1:7888/
launchctl setenv PIP_INDEX_URL http://127.0.0.1:7888/pypi/simple/
launchctl setenv GOPROXY http://127.0.0.1:7888/go,off
Works
  • Covers GUI apps
  • Persists across reboots
  • No TLS complexity
Doesn't catch
  • Truly bundled npm (ignores env)
  • Hardcoded registry calls
  • Non-proxy-aware tools
Option 3 — Network Backstop
pf packet filter (macOS) / iptables (Linux)
Intercept at the kernel network layer. Any TCP connection to port 443 destined for a registry host gets redirected to 127.0.0.1:7888 regardless of which tool made it, which config it reads, or whether it inherits env vars. The proxy user is exempted to prevent redirect loops. 
# /etc/pf.anchors/escrow-npm
rdr pass inet proto tcp from any to registry.npmjs.org port 443 \
  -> 127.0.0.1 port 7888

pass out quick proto tcp from any to registry.npmjs.org \
  port {80, 443} user 499  # proxy UID, not username

block return out inet  proto tcp from any to registry.npmjs.org port 80
block return out inet6 proto tcp from any to registry.npmjs.org port 80
block return out inet6 proto tcp from any to registry.npmjs.org port 443
Works
  • Catches all processes
  • VPN-compatible (redirects to localhost)
  • Blocks IPv6 bypass
Doesn't catch
  • IPs that rotated post-load
  • New CDN IPs (8s TTL!)
  • HTTP/3 / QUIC (UDP)
Option 4 — macOS Only, Complex
Network Extension (NETransparentProxyProvider)
Apple's System Extension framework can intercept flows before DNS resolution using the remoteHostname property on NEFilterSocketFlow. Truly hostname-aware, no IP rotation issue. But requires a provisioning profile with special entitlements, a System Extension bundle, and user approval.
Works
  • True hostname matching
  • Immune to CDN rotation
  • What Little Snitch uses
Problems
  • Cisco AnyConnect conflict
  • Email Apple for entitlement
  • Swift/ObjC only (not Go)
  • User approval required
Option 5 — Invasive
Global HTTPS_PROXY with TLS MITM
Set HTTPS_PROXY=http://127.0.0.1:7888 globally. All HTTPS traffic flows through the proxy, which decrypts, inspects, and re-encrypts using a locally-trusted CA. Hostname-aware, universal coverage — but requires installing a CA cert into the system keychain, Python's certifi bundle, Java's cacerts, Rust's webpki roots, and fighting every tool that pins certificates.
Works
  • Universal coverage
  • Hostname-aware
  • No pf needed
Problems
  • Requires MITM CA cert
  • 5+ trust store installs
  • Certificate pinning breaks
  • HTTP/3 still bypasses it

Why 100% is Hard

Even with every option above deployed simultaneously, there are structural gaps that are genuinely difficult to close without either imposing significant developer friction or becoming the kind of security tool that breaks things more than it protects them.

The gaps that remain
CDN IP rotation
pf resolves hostnames to IPs at rule-load time and stores them statically. proxy.golang.org has a TTL of 8 seconds. By the time a developer runs go get, the IP is almost certainly different from the one in the rule. The rule doesn't match; traffic goes direct.
HTTP/3 · QUIC
QUIC is UDP. pf redirects TCP. An HTTPS proxy speaks TCP. A package manager that negotiates HTTP/3 sends packets on UDP/443 that no TCP-based interception layer will ever see. Most current CLI tools don't use HTTP/3 — but that's changing as CDNs push it harder.
Bundled runtimes
VSCode bundles its own Node.js at /Applications/Visual Studio Code.app/Contents/Resources/app/node_modules/.bin/npm. When an extension triggers npm install, that bundled npm may read from ~/.npmrc — or it may not. Version differences, custom --userconfig flags, and extension isolation policies all affect this. pf catches it at the network layer regardless; env injection catches it if the bundled npm honours env vars.
VPN exclusion zones
Corporate VPNs often mark registry IP ranges as "direct" (bypassing the tunnel) in their split-tunnel config. Traffic to those IPs leaves the machine without touching the pf redirect or the proxy — and it's configured on the VPN server, not the developer's machine.
DNS-over-HTTPS
DNS interception (/etc/hosts, dnsmasq) is another potential interception layer. But Firefox, Chrome, and newer versions of curl can bypass the system resolver entirely using hardcoded DoH endpoints. For package managers specifically this isn't yet a concern — but it closes the DNS door for the future.
Hardcoded IPs
Malicious packages occasionally connect to exfiltration endpoints using hardcoded IP addresses, bypassing DNS entirely. A rule for registry.npmjs.org doesn't help if the package calls 104.16.0.34 directly. This is pf's home turf — and it still requires knowing the IP ahead of time.

The DNS TTL reality check

Here's what we measured when we did a fresh DNS lookup of every registry host, run twice five minutes apart:

proxy.golang.org TTL: 8s Google — IPs rotated between both lookups
repo.packagist.org TTL: 35s IP changed between runs
packagist.org TTL: 35s IP changed between runs
static.crates.io TTL: 231s Fastly — IP changed between runs
api.nuget.org TTL: 168s Azure — stable in practice
registry.npmjs.org TTL: 299s Cloudflare anycast — 12 IPs, all loaded by pf
pypi.org TTL: 48,557s Fastly — 4 stable IPs for ~13 hours

proxy.golang.org at 8 seconds means pf rules are stale almost immediately. Every go get is a coin flip on whether the redirect fires. Refreshing fw-enable via a launchd timer is the only mitigation short of a Network Extension.

The Layered Defence Model

No single layer reaches 100%. Three layers together reach the point where the remaining gap requires a deliberately adversarial package that does something unusual — and that's a useful place to be.

Layer CLI tools GUI apps Bundled runtimes Rogue scripts
1. Per-tool config
config write		 ✓ ✗ ~ ✗
2. Launch env vars
launchctl setenv		 ✓ ✓ ~ ✗
3. pf / iptables
fw-enable		 ✓ ✓ ✓ ~

The ~ on pf for rogue scripts is the CDN IP rotation issue. Mitigate it by refreshing pf rules on every network change event — a launchd job watching com.apple.network.change brings that closer to ✓.

What this model doesn't catch: a package that connects to an exfiltration server on an IP not in the ruleset, or a tool that uses HTTP/3 over UDP. For those, you need egress filtering at the network level (corporate firewall, DNS sinkhole, or a genuine Network Extension with Apple's blessing).

The Honest Conclusion

Perfect supply chain interception on a developer machine requires either becoming the OS (kernel extension, Network Extension with entitlements) or accepting that determined adversaries have escape routes. The good news is that real supply chain attacks don't require exotic evasion — they rely on developers installing a package without checking. Intercepting the happy path is achievable, and the happy path is where 99.9% of the risk lives.

The layered model above — per-tool config, launch environment injection, and a network-layer backstop — covers every ordinary installation scenario: npm install in a terminal, a VSCode extension auto-installing its dependencies, a CI job running in a local runner, a Zed plugin downloading a language server. That's the threat model worth defending.

You don't need to catch every possible evasion. You need to catch the attack that's actually happening — and today, that attack goes through npm install.

The remaining gap is worth documenting honestly so teams don't build false confidence into their security posture. Escrow is a layer, not a silver bullet. Treat it that way — and it's genuinely useful.
